			<p><!--SSW Code Auditor - Ignore next line(HTML)-->  
				Support emails arrive in the info mailbox. The info@ssw.com.au mailbox administrator will reply to the email from the user 
				with a request &quot;Please action&quot; (with the user CC'd) to 
				you as the Product Manager.
			</p>
			<p>
				Developers need to know allocate high priority bugs over low priority 
				ones. (look up &quot;<a target="_blank" href="../../Redirect/Google/Google.htm">Triage Bugs</a>&quot; on google). This is how you 
				<a title="Triaging is the process of pushing Bugs through their life cycle as well as reprioritizing Bugs according to user reports and as a result of Bugs being closed by developers" target="_blank" href="../../Redirect/Google/Google2.htm">triage</a> the email at 
				SSW:</p><ul>
					<li>LITTLE INCIDENTS&nbsp; - If it is a quick issue (i.e. &lt; 1/2 hour) 
					then action immediately 
					<ul>
						<li>examples are known issues (email using a KB link) or a quick BUG fix</li><li>reply using Incident PRO! (as an additional item in the current 
						release)</li></ul></li>
					<li>IMPORTANT - If it is a bigger issue (i.e. &gt; 1/2 hour) it is 
					allocated into the Public Folder 
					<ul>
						<li>examples are more complicated bugs </li>
						<li>move to Public Folder using Incident PRO!  usually to the 
						current release eg. Release09</li><li>Note: Reply to the customer using Client Diary Category &quot;Product - 
						Support - Product Suggestion</li></ul></li>
					<li>STANDARD - enhancements and suggestions (ensure enhancements are 
					approved)<ul>
						<li>examples are user interface improvements, changes to reports</li><li>move to Public Folder using Incident PRO!  usually to the next 
						release eg. Release10</li></ul>
					</li>
					<li>FUTURE FEATURES - things we don't need at SSW and probably will 
					never do<ul>
						<li>examples are adding security support to Data Renovator</li><li>move to Public Folder using Incident PRO!  to a folder named 
						zsFutureFeature</li></ul>
					</li>
					</ul>
					<p>If you are the primary developer and are overloaded (that is you are not able to action the Incident within 48 hours), you 
					get help from the secondary developer (or contact your manager). eg. I am 
					working on site for 3 weeks and have 10 customer emails that I cannot do.</p>
					<p>Note: a developer should not have any support emails from customers (in their inbox) that are older than 2 days.</p>
